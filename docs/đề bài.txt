1. Phân Tích Yêu Cầu Đề Bài (Topic 9: Unified E-Health Patient Records)
Đề tài này yêu cầu giải quyết bài toán "Đa hình thái dữ liệu" (Polyglot Persistence): Dữ liệu lịch sử bệnh án thì lộn xộn, đa dạng (NoSQL) nhưng lịch hẹn bác sĩ thì phải tuyệt đối chính xác (SQL).
4 Yêu cầu Kỹ thuật Cốt lõi bắt buộc phải có:
1. Chặt chẽ về Lịch hẹn (SQL Strict Consistency):
    ◦ Yêu cầu: Viết Stored Procedure xử lý việc đặt lịch.
    ◦ Điểm mấu chốt: Phải chứng minh được việc ngăn chặn "Double-booking" (hai người đặt cùng 1 giờ) bằng Transaction Isolation Level (ví dụ: SERIALIZABLE).
2. Linh hoạt về Hồ sơ (NoSQL Schema Evolution):
    ◦ Yêu cầu: Lưu lịch sử khám trong MongoDB.
    ◦ Điểm mấu chốt: Minh họa tính năng "Schema Evolution" – tức là thêm một trường mới (ví dụ: vaccination) vào các bản ghi mới mà không làm lỗi các bản ghi cũ, không cần chạy lệnh ALTER TABLE như SQL.
3. Truy vấn Phả hệ (SQL Recursive CTE):
    ◦ Yêu cầu: Sử dụng kỹ thuật Recursive Common Table Expression (CTE).
    ◦ Mục đích: Từ một bệnh nhân, truy xuất ngược lên cây gia phả để tìm cha mẹ/người thân (phục vụ phân tích di truyền).
4. Phân tích Dữ liệu (MongoDB Aggregation):
    ◦ Yêu cầu: Dùng Aggregation Pipeline (không dùng lệnh find đơn giản).
    ◦ Mục đích: Lọc và thống kê bệnh nhân theo chẩn đoán bệnh từ mảng dữ liệu phức tạp.

• Ví dụ: Khi 2 y tá cùng bấm lưu lịch hẹn cho 1 bác sĩ lúc 9:00 AM, hệ thống phải chặn 1 người lại. Nếu không có logic LOCK hoặc TRANSACTION, bạn sẽ bị trừ điểm nặng.

D. Vấn đáp Cá nhân (15%) - "Sửa code tại trận"
• Giảng viên sẽ yêu cầu bạn sửa code ngay lúc đó (ví dụ: "Đổi RANK() thành DENSE_RANK() xem kết quả khác gì?"). Nếu bạn không tự viết code mà phó mặc cho AI, bạn sẽ trượt phần này.
